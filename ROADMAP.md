# Encore 产品路线图（内部）

> 这份文档是对内的施工计划。对外愿景见 README.md。
> 原则：架构上留口，代码上不设计。每个阶段只做一件事。

---

## V0.1 — MVP：跑通链路

**目标**：验证"对话 → 分类 → 笔记"这条链路是否真的有用。

**范围**：
- [ ] **Encore Skill for Claude Code**：一个 `/encore` 命令，接收对话文本
- [ ] **意图分类器**：调用 LLM 做 few-shot 分类，输出 bug_fix / learning / idea
- [ ] **三类提取模板**：每种意图对应一个结构化提取 prompt
- [ ] **本地存储**：输出为 Markdown + YAML frontmatter，存入 `~/.encore/notes/`
- [ ] **内部 JSON Schema**：定义 `encore_note_v0.1.json` 数据契约，仅自己使用

**不做的**：
- 不做浏览器插件、IDE 插件
- 不做 Notion/Obsidian/Gist/Todoist 集成
- 不做自动触发、闲置检测
- 不做向量检索、记忆回响
- 不做多意图混合分类

**验收标准**：
- 在 Claude Code 中完成一次真实对话后，输入 `/encore`，能在 `~/.encore/notes/` 下生成一条结构化的 Markdown 笔记
- 连续使用一周，至少产生 10 条笔记，自己真的会回头翻看

---

## V0.5 — 抽象：从工具到数据标准

**前置条件**：V0.1 验证通过，自己已经用了至少 20 条笔记。

**范围**：
- [ ] **统一 JSON Schema**：基于 V0.1 的实践经验，正式定义 `encore_note_v0.5.json`
- [ ] **适配器模式**：输出端支持本地 Markdown + Notion API（仅两个适配器）
- [ ] **CLI 独立工具**：`encore` 命令行，不依赖 Claude Code Skill 即可独立调用
- [ ] **第二条输入通道**：支持 Cursor 或网页端手动粘贴对话文本

**不做的**：
- 不做浏览器/IDE 插件
- 不做自动触发
- 不做向量检索

**验收标准**：
- 同一个 `encore` CLI 既能存本地 Markdown，也能推送到 Notion
- 数据格式 V0.5 向下兼容 V0.1

## V1.0 — 产品化：多端覆盖

**前置条件**：V0.5 稳定运行，至少有两种输入端和两种输出端在正常使用。

**范围**：
- [ ] **Encore Browser Extension**：Chrome 插件，支持 ChatGPT / Claude Web 端一键归档
- [ ] **多输出端**：Todoist（待办事项）+ GitHub Gist（代码片段）
- [ ] **周报推送**：每周生成一份本周归档摘要
- [ ] **本地检索**：基于文件名和 frontmatter 的关键词搜索
- [ ] **首次对外发布**：公开发布 pip 包 + 文档站

**验收标准**：
- 浏览器插件可独立使用，不依赖命令行
- 每周推送的摘要能覆盖本周所有归档内容
- 有至少 5 个外部用户成功安装使用

---

## V2.0 — 记忆回响：让知识"活"过来

**前置条件**：V1.0 有稳定用户群，积累了足够的真实使用数据。

**范围**：
- [ ] **向量检索**：ChromaDB / pgvector 存储笔记 embedding
- [ ] **上下文注入**：新对话开始时，自动检索相关历史笔记并附在上下文中
- [ ] **语义搜索**：`encore search "HuggingFace 报错"` 返回语义相关笔记
- [ ] **知识图谱**：笔记间的关联关系可视化

**验收标准**：
- 搜索命中率：用户真实搜索 top-3 召回率 ≥ 80%
- 上下文注入后，用户不需要手动搜索即可获得历史提醒

---

## V3.0 — 协议化：Encore 成为标准

**前置条件**：社区有第三方贡献的适配器或输入端插件。

**范围**：
- [ ] **Encore Protocol Specification v1.0**：公开发布协议规范文档
- [ ] **Encore SDK**：pip 包 + npm 包，第三方可一键生成符合标准的数据
- [ ] **社区适配器生态**：接受第三方 PR 贡献新的输出端适配器
- [ ] **Encore Hub**：适配器和输入插件的发现/安装目录
